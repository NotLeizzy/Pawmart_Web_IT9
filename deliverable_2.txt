System Description
Overview:
PawMart is a web-based online pet shop system. The system is designed to provide customers with an efficient platform for browsing pet products, checking product availability, and placing online orders.
The system also assists administrators in managing products, inventory, and customer orders through a centralized dashboard. To improve efficiency and scalability, the system applies appropriate data structures and algorithms for searching, sorting, inventory tracking, and order processing.
 
Target Users:
Pet owners
Pet shop owners

Core Functionalities:
1.     Manage products, categories, and inventory
2.     Search and filter pet products efficiently
3.     Process customer orders
4.     Monitor real-time inventory updates
5.     Prioritize and manage orders
6.     Generate sales and inventory reports

Baseline System Design
Data Structures Used:
Arrays/Lists for storing products, users, and orders retrieved from the database
Relational Database (MySQL) for centralized data storage
Sequential Records for order processing and inventory updates

Algorithms Used:
Linear Search for searching products and orders
Basic Sorting mechanisms for arranging products by name or date
Sequential Processing (First-Come, First-Served) for order handling

Workflow:
Product, user, and order information are retrieved from the database.
Customer orders are processed sequentially.
Inventory records are updated after each transaction.
Product searches scan records using standard database queries.
Reports are generated by scanning the entire dataset.

Identified Inefficiencies:
Search operations become slower as product records increase
Full-table scans increase database processing time
Repeated retrieval of similar data causes redundant database queries
Report generation becomes inefficient for large datasets
System scalability decreases as transactions and inventory grow

Optimized System Design
Data Structures Used
AVL tree:
A self-balancing Binary Search Tree (ProductBST) that arranges products alphabetically by name. By keeping nodes balanced through rotations, lookup and insertion stay bounded to O(log n) worst-case complexity, avoiding list degradation.

Priority Queue (Max-Heap):
Manages order processing by scheduling and prioritizing urgent and high-value orders.

Hash Table:
A custom chaining hash table implemented to analyze and model constant-time database reference patterns, while production lookup utilizes optimized database indexes.

Arrays/Lists:
Used to hold collections of products and orders prior to structuring.

Algorithms Implemented:
BST Search & Insertion: Traverses tree node relations case-insensitively (O(log n) average and worst case) to insert and retrieve products.
Greedy Max-Heap Scheduling: Computes numerical priority weights based on order status urgency and order value, then extracts the highest-priority orders in O(log n) time.
Merge Sort: A stable, recursive divide-and-conquer sorting algorithm (O(n log n) complexity) to arrange products and transactions.
Polynomial Rolling Hashing: Converts keys to hash indices with chaining to simulate immediate O(1) average-time lookups.

Improvements:
Product Search: Optimized from linear scans to logarithmic traversals using BST Search.
Order Scheduling: Priority-based order scheduling replaces standard FCFS processing using a Max-Heap.
Lookup Time: Database lookup latency is significantly reduced via Eloquent indexing, and simulated using a custom Hash Table.
Sorting Efficiency: Improved using a stable Merge Sort service.
Inventory Updates: Dynamic stock calculations benefit from optimized database/caching lookups.

Data Structures and Algorithms Justification
Binary Search Tree (BST): Organizes products by category and name for faster searching and sorting, keeping paths balanced to guarantee logarithmic search heights.
Hash Table: Provides near constant-time lookup for products and users.
Priority Queue (Heap): Prioritizes urgent and earlier orders efficiently.
Merge Sort: Ensures stable and efficient sorting of products and transactions.
Greedy Algorithm: Selects urgent orders first to improve scheduling efficiency.

Implementation Disclosure (Corrected):
BST (AVL Tree), heap, merge sort, and hash table are implemented from scratch in PHP. There is no custom binary search; AVL BST search is utilized instead. The hash table is written from scratch to model and analyze constant-time access patterns, while the framework uses Eloquent indexing.

Implementation Challenges and Solutions
Challenges:
1. Manually Implementing BST and Heap: Resolving left/right height balance offsets and parent-child swap indices. Sourced via rotational recursive subtrees and swap helpers.
2. Synchronizing Real-time Inventory: Ensuring database records match fast in-memory hashes immediately. Solved by binding transactional DB updates together with in-memory hashing updates.
3. Integrating UI Modules: Merging Blade views with memory arrays without breaking paginated index files. Sourced via dynamic LengthAwarePaginator custom wrapping in controllers.
4. Category Relationships: Managed adjacency list tree representations to prevent memory leak degradation.
